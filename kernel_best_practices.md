# Kernel Best Practices

Optimization playbook for the tsgu CUDA kernels, to be applied **after** the migration
lands: each section is a priority-ordered list of techniques for one kernel family from
[spec/kernels.md](spec/kernels.md), with sources and decision rules for where each
technique wins. Every claim below traces to a cited source or to the verified hardware
table; unsourced design reasoning is explicitly marked as such.

Provenance: assembled 2026-07-17 from a 33-agent web-research run (20 searchers × 3
angles per kernel, per-kernel synthesis, completeness critique + gap-fill), plus an
8-agent curated-source round on the PTX/microarchitecture layer (WebFetch-only,
fetch failures flagged inline rather than filled from memory) with two claims
settled by local SASS probes on the dev GPU. Spec
acceptance bars (beat cuSPARSE / pytorch_scatter, deterministic-path requirement,
no dense materialisation, O(nse) workspace) are taken as fixed constraints, not
re-litigated here.

## SDDMM (sampled dense-dense matmul)

**What dominates performance.** SDDMM at our shapes is memory-bound gather: every specified entry (i,j) reads two dense rows G[i,:] and B[j,:] of length p and writes one scalar, so arithmetic intensity is ~1 FLOP per 4 bytes read and the ~192 GB/s DRAM of the A1000 is the wall. The three levers, in order, are (1) making the row gathers coalesced and vectorized, (2) not re-reading rows that many entries share (cache/shared-memory reuse — critical with a small L2), and (3) keeping enough independent reductions in flight that the memory latency is hidden despite the dot-product's serial tail. For small p the reduction itself is nearly free and everything is traffic; only at large p does reduction parallelism (splitting p across more lanes/warps) become a first-order concern. fp32 is the performance path; fp64 (1/64 rate on GA10x sm_86) will additionally be FMA-limited at larger p, which makes the traffic optimizations below matter even more since there is no compute headroom to hide behind.

### Best practices, in priority order

1. **Warp-per-entry mapping with shuffle reduction over p.** Assign one warp (or sub-warp) per specified entry; lanes stride contiguously across the p dimension of G[i,:] and B[j,:], accumulate lane-local FMAs in registers, then reduce with `__shfl_down_sync` — 5 shuffle steps, no shared memory, no `__syncthreads`, and every output entry has exactly one owner (atomic-free and deterministic by construction). This is the Sputnik/FeatGraph scheme and matches our current plan; work per entry is uniform (always p), so load balance is inherently good regardless of row-length skew. For small p, size the sub-warp as the smallest power of two ≤ p (e.g., 8 lanes for p=8) rather than wasting a full warp — the AMD SpMV note's power-of-2 thread-allocation rule applies directly to the reduction width ([Sputnik](https://arxiv.org/pdf/2006.10901), [AMD lab notes](https://gpuopen.com/learn/amd-lab-notes/amd-lab-notes-spmv-docs-spmv_part1/), [warp-shuffle reduction](https://medium.com/@aditi.sikarwar25/parallel-reduction-in-cuda-warp-shuffle-instructions-register-level-communication-3f1e3dde4ae5)).

2. **128-bit vectorized loads with aligned row bases.** Load G/B rows as `float4` (fp32) or `double2` (fp64) to cut load-instruction count 4× and emit LDG.E.128; on a bandwidth-starved part this is the single cheapest win. Requires 16-byte-aligned row base pointers — pad the leading dimension of G and B to a multiple of 4 floats, or use Sputnik's reverse-offset alignment trick (round the start address down to the alignment boundary and mask the over-read lanes), which Sputnik credits for raising memory-throughput utilization from ~30% to ~70% on memory-bound sparse kernels. Watch register pressure: vector loads plus 2–4 accumulators per lane is fine, but don't stack this on top of aggressive unrolling without checking occupancy ([NVIDIA vectorized-access pro tip](https://developer.nvidia.com/blog/cuda-pro-tip-increase-performance-with-vectorized-memory-access/), [Sputnik](https://arxiv.org/pdf/2006.10901)).

3. **Multiple entries per warp for small p (warp-merged / interleaved reduction).** When p ≤ 32, a warp-per-entry wastes lanes and the shuffle tree dominates. Process 2–4 entries per warp with interleaved accumulators, then do one segmented shuffle reduction — the PRedS IIR/WMT scheme reports 1.20× (V100) to 2.09× (RTX 2080) at reduction dim 32 from exactly this change, and it doubles the number of independent memory streams per warp, which is how latency gets hidden on a low-bandwidth Ampere part. Conversely, when p ≫ 32, go the other way: split p across multiple warps hierarchically (WMT) instead of looping serially in one warp. Decision rule: entries-per-warp = max(1, 32/p) for p ≤ 32; warp-per-entry with a serial p-loop for 32 < p ≲ 256; multi-warp split-p above that ([PRedS](https://nicsefc.ee.tsinghua.edu.cn/nics_file/pdf/1677968f-e94f-4c11-b5de-15700bd10d79.pdf)).

4. **Route dense-row reads through the read-only cache.** Mark G and B pointers `const __restrict__` and/or use `__ldg()` so gathers go through the read-only/texture path of sm_86's unified 128 KB L1, keeping ordinary L1 free for indices. Gather-like sparse kernels measured 71–81% read-only-cache hit rates vs 52–57% through the default path — on a GPU with a small L2, this on-SM reuse is where repeated G[i,:] reads (many entries per row i in CSR order) actually get caught ([PyTorch #19165](https://github.com/pytorch/pytorch/issues/19165), [Ampere tuning guide](https://docs.nvidia.com/cuda/ampere-tuning-guide/index.html)).

5. **Shared-memory tiling of G rows for large p — the planned second path.** When p is large and entries are processed in CSR row order, stage G[i,:] into shared memory once per block and let all entries of that row reuse it, converting O(entries_in_row × p) DRAM reads into O(p): footprint is block_dim × p × 4 bytes, i.e., keep p ≲ 128 at 32-wide tiles (~4 KB/block, comfortable in sm_86's shared-memory limit (99 KB per block opt-in, 100 KB carveout per SM)). This is FeatGraph's SDDMM strategy and is worth the most on the A1000 precisely because DRAM is slow relative to shared memory (1–2 cycles vs 100+). Threshold between path 1 and this path should be picked empirically around p ≈ 128–256; below that, the read-only cache usually captures the reuse for free ([FeatGraph](https://arxiv.org/pdf/2008.11359), [Sputnik](https://arxiv.org/pdf/2006.10901)).

6. **Overlap global loads with compute via `cp.async` k-tiling in the large-p path.** Inside the shared-memory path, iterate p in tile_k chunks of 8–32, using Ampere's `memcpy_async`/`cp.async` to prefetch tile k+1 of G (and optionally B) while reducing tile k — this hides the 100–400-cycle global latency without burning registers on manual double-buffering and is one of the few sm_86-specific features that directly helps a bandwidth-bound kernel ([Ampere tuning guide](https://docs.nvidia.com/cuda/ampere-tuning-guide/index.html), [GPU matmul tiling](https://indii.org/blog/gpu-matrix-multiply-tiling/)).

7. **Prefer the CSR entry ordering; treat COO as the one-shot/cheap-construction case.** CSR keeps same-row entries adjacent, which is what makes G-row reuse (items 4–5) work and avoids a per-entry row lookup; measured 1.2–1.5× over COO for SDDMM, and it is what cusparseSDDMM (our baseline) consumes. Decision rule: if the pattern is reused across calls (our solver backwards are), convert once via the existing tsgu::coo2csr and run the CSR kernel; keep a direct COO kernel only for single-shot patterns where the O(nnz) conversion wouldn't amortize. Row/entry sorting or merge-path balancing (GE-SpMM's 1.41–1.81× over cuSPARSE) is only needed if we ever adopt a block-per-row schedule — the warp-per-entry mapping makes it unnecessary ([cuSPARSE docs](https://docs.nvidia.com/cuda/cusparse/), [GE-SpMM](https://arxiv.org/pdf/2007.03179)).

8. **Fuse the negate-and-scale epilogue into the reduction registers.** Apply `−α·dot` on the lane-0 accumulator immediately before the single store — zero extra workspace, zero extra traffic, and it removes a whole elementwise kernel from every backward. Batched variants should fold the batch offset arithmetic into the same store rather than launching per-batch ([Sputnik-style fused epilogues](https://arxiv.org/pdf/2006.10901)).

9. **Tensor-core / hybrid paths: only behind a density gate, and probably never for us.** Tensor-core SDDMM (FlashSparse, Libra, block-reordering schemes) posts large speedups (up to 10.38× over TC baselines) but only on patterns with dense sub-blocks; Libra's empirical gate is per-block utilization η = NNZ_block/(m×n) ≥ 0.1875, below which CUDA cores win. Our patterns come from arbitrary solver sparsity and are unstructured, and the bar is CUDA-core cusparseSDDMM — so document the η threshold, but do not build a TC path unless profiling shows real blocky inputs; the reordering preprocessing also adds O(nnz) workspace and complexity ([Libra](https://arxiv.org/abs/2506.22714), [FlashSparse](https://arxiv.org/abs/2412.11007), [block-reordered SDDMM](https://link.springer.com/article/10.1007/s11227-026-08606-2)).

### Deterministic path

The warp-per-entry design is deterministic for free: each output entry is written exactly once by its owning warp, and the shuffle tree is a fixed-shape reduction, so results are bitwise-reproducible run-to-run for a given launch configuration. Non-determinism only enters if large p forces split-p partial sums to be combined across blocks with `atomicAdd` — that is the fast path there (atomics measured 1.5–2× faster than tree combination at small scales), and the deterministic alternative is a fixed-order two-stage reduction: stage partials to an O(nnz × splits) workspace (bounded, since splits is a small constant) and combine in a second kernel in index order. Keep the launch geometry (entries-per-warp, tile_k, split factor) a pure function of the input shape, not of occupancy heuristics queried at runtime, or "deterministic" silently becomes "deterministic per machine state" ([NVIDIA CCCL determinism](https://developer.nvidia.com/blog/controlling-floating-point-determinism-in-nvidia-cccl/), [Quansight/PyTorch BSR](https://quansight.com/post/pytorch-2-1-quansights-improvements-to-bsr-sparse-matrix-multiplication/)).

## Segmented logsumexp (+ bidirectional)

**What dominates performance.** This kernel is memory-bandwidth-bound on the RTX A1000 (~192 GB/s, small L2): each extra pass over `values` costs a full global-memory scan, so pass count is the first-order knob. Second-order costs are (a) atomic contention on the per-segment accumulators — especially for the bidirectional variant, where at least one of the two index directions is unsorted relative to the storage order — and (b) the fp32 exp/rescale arithmetic itself, which Ampere executes on the non-matmul pipe and which FlashAttention-2 showed must be minimized per segment, not per element. Skewed segment-length distributions (a few dense rows among many near-empty ones) add a load-balancing hazard on the A1000's small SM count.

1. **One-pass online logsumexp for the fast path; two-pass for the deterministic path — decide by mode, not by benchmark.** The online algorithm (Milakov & Gimelshein) carries a running pair `(m, s)` per segment and rescales on a new max: `s = s * exp(m_old - m_new) + exp(v - m_new)`, then `lse = m + log(s)`. It cuts global traffic roughly in half versus two-pass and is the natural fit for the bidirectional variant (one traversal feeds both row and col accumulators), but the rescale order is thread-schedule-dependent, so it is inherently the non-deterministic path. Measured regimes: negligible gain at 2^11–2^15 elements, real gains above ~2^16 — exactly our nnz-scale sparse workloads. ([arxiv.org/abs/1805.02867](https://arxiv.org/abs/1805.02867), [hugo.rosenkranz — online softmax](https://medium.com/@hugo.rosenkranz/gpu-kernel-optimization-online-softmax-a92b8ebe24e3))

2. **Load-balance by element, not by segment (MGPU-style values-per-thread).** Assign each thread a fixed VT of 4–8 values from the flat `values` array and detect segment boundaries inline, rather than mapping threads/warps to rows. This makes throughput independent of segment geometry — critical for skewed rows on a small-SM GPU — and MGPU reports ~20% gains for large segments by branching to a plain tile-wide reduction when a tile is single-segment. Tune VT ∈ {4, 8, 12} empirically on sm_86; larger VT increases reuse but risks register spill (~24–32 registers/thread budget for value slots plus `(m, s)` accumulators). ([moderngpu.github.io/segreduce.html](https://moderngpu.github.io/segreduce.html))

3. **Hierarchical reduction: warp shuffles first, shared memory once, atomics last.** Reduce within warps using `__shfl_down_sync` (1–2 cycle latency, no bank conflicts, no `__syncthreads`); the online merge of two partials is `m = max(m1,m2); s = s1*exp(m1-m) + s2*exp(m2-m)` and applies identically at warp, block, and grid level. The softmax worklog progression is instructive: naive 124 ms → online 88 ms → shared-memory reduction 6.6 ms → shuffle-based 5.2 ms — the hierarchy, not the online trick alone, delivers the order of magnitude. When a segment fits within one warp (common for sparse rows), `cub::WarpReduce::HeadSegmentedSum/TailSegmentedSum` with head flags generates the optimal shuffle code directly; sm_86 also has native `__reduce_add_sync`-family warp reductions for 32-bit operands. ([maharshi — softmax worklog](https://maharshi.bearblog.dev/optimizing-softmax-cuda/), [cub::WarpReduce](https://nvidia.github.io/cccl/unstable/cub/api/classcub_1_1WarpReduce.html), [Ampere tuning guide](https://docs.nvidia.com/cuda/ampere-tuning-guide/index.html))

4. **Warp-aggregate atomics by segment key before touching global memory.** Before any atomic, use `__ballot_sync`/`__match`-style voting to find peer threads holding the same segment id, tree-reduce the peer group via shuffles, and issue one atomic per warp per segment instead of one per element. Gains are largest for sorted/semi-sorted keys (our CSR-major direction) and vanish for random keys — which is precisely the cross direction of the bidirectional variant, so expect the col-side atomics to dominate and budget accordingly. ([NVIDIA — voting and shuffling to optimize atomics](https://developer.nvidia.com/blog/voting-and-shuffling-optimize-atomic-operations/))

5. **Float atomic max via integer reinterpretation, never a CAS loop.** For the running-max update, cast with `__float_as_int`/`__int_as_float` and use native integer `atomicMax` (valid for the non-negative-ordered int encoding; handle the sign trick for negatives). This is roughly 10x faster than a float CAS spin loop and avoids CAS-induced warp divergence when key order correlates with data order. ([NVIDIA forums — atomicMax for float](https://forums.developer.nvidia.com/t/cuda-atomicmax-for-float/194207))

6. **Stage aggregation through shared-memory atomics; one global atomic per block per segment.** Shared-memory atomics on Ampere run ~10 cycles uncontended versus ~30–50+ for global atomics with coherency traffic; aggregate block-local `(m, s)` partials in an 8–32 KB shared workspace, then merge to global with a single fused update per (block, segment). Pad shared arrays to break 32-way bank conflicts when the reduction width exceeds one warp. This staging is fast-path only — fetch-and-op ordering is non-deterministic. ([supercomputingblog — atomics performance](https://supercomputingblog.com/cuda/cuda-tutorial-5-performance-of-atomics/), [arxiv 2503.17893 — shared-memory atomic bottlenecks](https://arxiv.org/pdf/2503.17893))

7. **Minimize rescale operations — at most ~one `exp(Δm)` per segment partial, not per element.** FlashAttention-2's core lesson: non-matmul fp32 FLOPs are ~16x more expensive than tensor-core FLOPs on Ampere, and its rescale-deferral rewrite bought 2x over FA1. For us: accumulate within a thread's VT chunk against the chunk-local max, and only rescale when merging partials up the hierarchy; for the bidirectional variant, batch the row- and col-side rescales rather than interleaving them per element. ([hazyresearch — FlashAttention-2](https://hazyresearch.stanford.edu/blog/2023-07-17-flash2))

8. **fp32 accumulation with max-shift is non-negotiable for fp16/bf16 storage.** Convert loaded values to fp32 immediately and always exponentiate shifted values `exp(v - m)` so intermediates stay in (0, 1]; raw `exp` overflows fp16 already at v ≈ 11–12 (fp16 max ~65k). The identity `lse = max + log(Σ exp(v - max))` is mathematically lossless, so the shift costs nothing in accuracy. ([gregorygundersen — log-sum-exp trick](https://gregorygundersen.com/blog/2020/02/09/log-sum-exp/))

9. **Backward: recompute, don't materialize; prefetch with `cuda::memcpy_async`.** The backward `exp(v - lse[seg]) * gout[seg]` is embarrassingly parallel with no reductions — one gather of `lse`/`gout` per value, no atomics, so it is deterministic for free in both modes. Follow the FlashAttention pattern of recomputing `exp(v - lse)` from stored `lse` rather than saving per-value softmax outputs (on a 192 GB/s part, recompute beats the extra O(nnz) write+read), and use sm_86 `cuda::memcpy_async` to prefetch value chunks into shared memory while prior results drain. ([stephendiehl — FlashAttention line by line](https://www.stephendiehl.com/posts/flash_attention/), [Ampere tuning guide](https://docs.nvidia.com/cuda/ampere-tuning-guide/index.html))

10. **Bidirectional traversal: sorted direction atomic-free, cross direction atomic.** With CSR-sorted values, the row direction has contiguous segments — reduce it with the shuffle/segmented-scan machinery of items 2–3 and no atomics; the column ids are effectively random, so the col accumulators must go through the warp-aggregated shared→global atomic chain of items 4–6. Don't force both directions through one strategy: the hybrid (one-pass per warp on the sorted side, atomic merge on the unsorted side) is the documented pattern for unsorted segment keys. ([dcbaslani — faster softmax kernel](https://medium.com/@dcbaslani/beating-pytorch-writing-a-faster-softmax-kernel-in-cuda-0d0a237cda57))

**Deterministic path.** Run two-pass over sorted segments: pass 1 computes per-segment max (O(rows) workspace, `cub::DeviceSegmentedReduce`-style tile reduction), pass 2 accumulates `Σ exp(v - max[seg])` with fixed-order warp tree reductions (`__shfl_sync` with colocated threads, stride halving — 99.3% branch efficiency vs 74% for the naive tree), then `lse = max + log(sum)`. No floating-point atomics anywhere and no inter-block fetch-and-op, which is what actually guarantees bitwise reproducibility across runs; expected cost is ~30% slower forward, under 2x in typical sparsity regimes. The backward kernel is shared between both modes since it needs no reductions. ([CUDA MODE lecture 9](https://christianjmills.com/posts/cuda-mode-notes/lecture-009/), [tensorflow#39751 — atomic non-determinism](https://github.com/tensorflow/tensorflow/issues/39751))

## SpMM / SpMV (CSR + COO, batched)

**What dominates performance.** On the RTX A1000 this op is memory-bandwidth bound, full stop: at ~192 GB/s every byte of index and value traffic is precious, and fp64's 1/64 rate never matters because we run out of bandwidth long before we run out of FLOPs. The two failure modes to engineer against are (1) wasted bandwidth — uncoalesced dense-B loads, int64 indices, redundant index re-reads across the batch — and (2) load imbalance on skewed (power-law) row distributions, where classical one-thread-per-row CSR kernels degrade to 0.3–0.5× while merge-path holds a 1.00× floor. The small L2 means working sets must be managed explicitly (tiling, persistence hints) rather than trusted to the cache.

1. **Merge-path workload partitioning (the default scheduler).** Use Merrill & Garland's 2D binary search over the (row-offsets, nnz) merge list so each thread/warp gets an equal share of nonzeros *plus* row-boundary events; this is what makes performance robust on skewed rows and is our stated edge over cuSPARSE. Reported results: 53% geomean speedup over cuSPARSE with 237% peak on short-row matrices, a guaranteed 1.00× minimum vs. row-split's 0.3–0.5× worst case, and ~7× instruction-level parallelism in the reference kernel. Decision rule: merge-path is the safe default for unknown/skewed structure; plain row-split (CSR-vector) wins on regular, uniformly long rows because it skips the partition-search overhead. ([Merrill & Garland merge-based SpMV](https://lukeo.cs.illinois.edu/files/2015_DaBaMeOlGa_merge.pdf))

2. **Coalesced, vectorized dense-B access.** Keep B row-major so consecutive threads in a warp read consecutive elements of the same B row, and issue 128-bit vectorized loads (`float4` / `double2` → LDG.128) instead of scalar 32-bit loads; the classic worklog measures coalescing alone taking effective bandwidth from 15 → 110 GB/s. When staging B tiles through shared memory, store them transposed so reads back out are LDS.128, not scalar LDS. Design-principles work reports 4.1× peak / 31.7% geomean from combining row-major coalesced access with merge-path balancing — these two items compound. ([siboehm CUDA MMM worklog](https://siboehm.com/articles/22/CUDA-MMM), [Yang et al., Design Principles for SpMM](https://people.eecs.berkeley.edu/~aydin/Yang2018EuroParallel.pdf))

3. **int32 indices everywhere; COO as 12-byte nonzeros.** Index traffic is pure overhead on a 192 GB/s part, so int32 halves it: an fp32 COO nonzero is (int32 row, int32 col, fp32 val) = 12 bytes. This is a differentiator vs. cuSPARSE's COO handling and directly serves the project's int32-COO goal. Keep CSR for regular/dense-row patterns (row-pointer indirection amortizes well there); prefer COO + segmented reduction for highly irregular structure. ([MERBIT](https://arxiv.org/html/2605.07391))

4. **Native batching: one kernel launch, blocks indexed by (batch, row-tile).** Launch a single kernel over the whole batch with `blockIdx.z` (or a flattened index) selecting the batch element, rather than looping cuSPARSE calls — this is the vendor gap we're exploiting. Batched SpMM reports up to 9.27× at batch size 64 and 6.09× at batch 512 vs. repeated non-batched invocations, from killed launch overhead, SM utilization rising 35%→87%, and L2 reuse of shared structure across batch elements. Within a block, assign 16–32 threads per row/nonzero depending on the dense column count p; cache-block the output when it exceeds shared memory. ([Batched SpMM for GCNs, ICS 2019](https://arxiv.org/pdf/1903.11409))

5. **Row-length-bucketed hybrid dispatch (CSR path).** Where row-length statistics are available (they fall out of CSR row pointers, or an O(rows) histogram computed once per matrix and reused across the batch and across Krylov iterations), dispatch per bucket: warp-per-row (CSR-vector) for rows above ~16–32 nnz, thread-per-row or merge-path for the short-row remainder — CSR-adaptive style hybrids report up to 424% over pure CSR-vector. Launch separate kernel variants per bucket rather than branching on row length inside a warp (intra-warp divergence costs ~2×). Acc-SpMM's trigger is a useful threshold: rebalance only when max_nnz/avg_nnz > 8, so already-uniform matrices skip the machinery. ([CSR-adaptive hybrid](https://www.researchgate.net/publication/284746100_Atomic_reduction_based_sparse_matrix-transpose_vector_multiplication_on_GPUS), [Acc-SpMM](https://arxiv.org/html/2501.09251))

6. **Atomic-free segmented reduction for the COO path.** With COO sorted by row (a sort we do once at tensor construction, O(nnz log nnz), amortized over every forward/backward and every Krylov iteration), compute one partial product per thread, reduce within warps via `__shfl`-based scans, then finish row segments with a block-level prefix sum — no atomics at all, deterministic by construction, one coalesced global write per block. This is the moderngpu segmented-reduction pattern and should be the backbone of the deterministic path. ([Modern GPU: segmented reduction](https://moderngpu.github.io/segreduce.html))

7. **Double-buffered `cp.async` prefetch (sm_80+).** Ampere's async global→shared copies (`memcpy_async` / cooperative-groups pipelines) let the next tile's indices and B values stream in while the current tile computes; reported gains are 1.06–1.16× over non-pipelined variants and ~20–30% fewer stall cycles, with Acc-SpMM's "least-bubble" double buffer as the reference design. Keep a prefetch-to-use distance of ~3–5 loads and watch register pressure — the second buffer's register fragments can cost occupancy. A realistic bandwidth-utilization target on this class of part is 50–60%. ([Acc-SpMM](https://arxiv.org/html/2501.09251))

8. **L2 persistence for reused sparse structure (Ampere access-policy window).** For batched ops and Krylov solvers the sparsity structure is identical across iterations, so pin the index arrays (CSR row pointers / COO rows) into a carved-out L2 region via the CUDA access-policy-window API; reported effect is a 30–50% cut in DRAM traffic for index arrays. Setup costs ~100–200 µs, so enable it only when structure is reused (batch > ~10 or inside an iterative solve) — set it up stream-side at op entry, never with a host sync in the hot path. ([NVIDIA Ampere Tuning Guide](https://docs.nvidia.com/cuda/ampere-tuning-guide/index.html))

9. **Register tiling + unrolling for ILP.** Within each merge-path segment, `#pragma unroll` the inner loop so 3–7 loads are in flight per thread and accumulate 4–8 rows' worth of partial dot products in registers before touching shared memory; this latency-hiding is what turns 1–2 sustained flops/byte into ~5–7 on a low-bandwidth part. sm_86 has a 256 KB register file per SM — tune the unroll factor against spills (`-maxrregcount` / launch-bounds sweeps), and note the warp-coarsening variant (4 warps cooperating per output tile) trades a little occupancy for less per-thread index-decode work when p is large. ([Merrill & Garland](https://images.nvidia.com/events/sc15/pdfs/sc15-Merge-Based-Parallel-Sparse-Matrix-Vector-Multiplication-merrill.pdf), [cuTeSpMM](https://arxiv.org/html/2504.06443v1))

10. **Cheap runtime algorithm selection.** Rather than one kernel for all inputs, pick the variant from three properties computable in O(rows) without host syncs (or cached at tensor construction): average nnz/row, row-length variance, and dense column count p. DA-SpMM's heuristic selector gets 1.26–1.37× over the best single algorithm on average; for us this is the dispatch layer that chooses merge-path vs. row-split, COO vs. CSR path, and fast vs. deterministic variant behind the single torch.library op. ([DA-SpMM, DAC 2022](https://arxiv.org/pdf/2202.08556))

11. **Tensor cores: only above a density threshold.** Hybrid TCU/CUDA-core schemes (Libra, RSH-SpMM) pay off only when tiles are dense enough — on consumer Ampere the reported break-even is ~30–40% column density, far above typical Krylov-system sparsity, and SpMV (p=1) can't use TCUs at all. Skip TCU paths for v1; revisit only if profiling shows dense-clustered inputs, and note the enabling preprocessing (affinity-based reordering, ~1.25–1.91× in Acc-SpMM/RSH-SpMM) is an offline, structure-permuting step that doesn't fit our drop-in-op contract. ([Libra](https://arxiv.org/html/2506.22714v2), [RSH-SpMM](https://arxiv.org/html/2603.08734))

**Deterministic path.** Ship the sorted-COO segmented-reduction kernel (item 6) as the deterministic variant: fixed reduction tree, no atomics, one coalesced global write per block. The fast path may use atomics for rows split across blocks, but stage them in shared memory first — shared-memory atomics run ~10× faster than global by shrinking the contention scope — and spill only one global atomic per block per output row. Measured trade-off: atomic variants are ~1.4× slower on dense/regular rows but win on highly skewed distributions, so the item-10 dispatcher should route regular matrices to the deterministic kernel even in fast mode (it costs nothing there) and reserve the atomic path for skew. ([FlashSketch](https://arxiv.org/pdf/2602.06071), [SMASH](https://arxiv.org/pdf/2105.14156))

## Triangular solve (SpSM)

**What dominates performance.** SpSM is latency- and dependency-bound, not FLOP-bound: arithmetic intensity is low (~1:8 FLOPs/byte) and the solve order is constrained by the dependency DAG of the sparsity pattern, so on the A1000's ~192 GB/s bandwidth and small L2 the twin costs are (a) inter-block synchronization between dependency levels and (b) redundant global-memory traffic for the RHS/solution vectors. Realistic throughput on this class of hardware is ~30–50 GFLOP/s versus 200+ on an A100, so every practice below is about shortening the critical path, keeping synchronization on-chip, and amortizing memory latency across the k RHS columns. The analysis phase is off the hot path by design — it is computed once per sparsity pattern and cached on the descriptor — so it can afford real work (BFS, coloring, dependency pruning) that a one-shot solver could not.

1. **Cache the level schedule on the descriptor; build it on-GPU.** Compute the level assignment with a parallel BFS / level-synchronous topological sort over the dependency DAG in a single sweep of the CSR structure — no CPU roundtrips, no host syncs — and store levels + dependency edges in an O(rows + nnz) device workspace on the descriptor, reused across every solve call with the same pattern (the cuSPARSE `csrsm2_analysis` → `csrsm2_solve` ×many pattern). Even a fast analysis costs 1–5 ms on large matrices, so budget for ≥10–20 solve calls before ROI — exactly what iterative/rsample loops give us. During analysis, prune redundant dependency edges (>90% elimination is reported), which directly reduces synchronization points in the solve phase. ([GPU level-set analysis](https://www.researchgate.net/publication/326854116_A_New_GPU_Algorithm_to_Compute_a_Level_Set-Based_Analysis_for_the_Parallel_Solution_of_Sparse_Triangular_Systems), [dependency sparsification](https://arxiv.org/abs/2012.06959), [cuSPARSE csrsm2](https://docs.nvidia.com/cuda/archive/12.1.0/cusparse/))

2. **Batch all k RHS columns per level in one launch.** Process every RHS column for a level inside a single kernel by assigning warp teams (or thread groups) per column, instead of dispatching column-by-column. This divides the number of global synchronizations by k and amortizes the memory latency of loading matrix rows across columns — the highest-leverage lever on bandwidth-starved hardware, and the main reason SpSM should beat k independent SpSV calls. RHS workspace is O(rows × k), within budget. ([batched algorithms](https://arxiv.org/html/2509.03015v1), [cublasDtrsmBatched pattern](https://scikit-cuda.readthedocs.io/en/latest/generated/skcuda.cublas.cublasDtrsmBatched.html))

3. **Sync-free spin-flag scheduling vs. barriered level-sets — pick by DAG shape.** The synchronization-free family (Liu et al. 2016; CapelliniSpTRSV) replaces inter-level barriers with self-scheduling: each row spin-polls per-dependency ready flags and proceeds the moment its inputs are done. Reported wins: 2.14–2.3× over cuSPARSE (up to 5.95× fp32), and CapelliniSpTRSV 4× over cuSPARSE across 245 matrices with zero preprocessing, directly on CSR. Decision rule: sync-free wins on deep, narrow DAGs (many levels, few rows per level) where barrier overhead dominates; barriered level-set execution wins on shallow, wide DAGs where each level saturates the GPU and the barrier cost is amortized — and since our analysis is cached, the level-set preprocessing cost (the usual argument against it) is already sunk. Ampere's independent thread scheduling (sm_70+) makes spin-polling viable without warp-wide stalls; keep flags compact (~1 byte per dependency) and in shared memory where the subdomain allows. ([Liu et al. 2016](https://www.ssslab.cn/assets/papers/2016-liu-sptrsv.pdf), [CapelliniSpTRSV](https://dl.acm.org/doi/10.1145/3404397.3404400), [spin-loop scheduling costs](https://people.ece.ubc.ca/aamodt/publications/papers/eltantawy.hpca2018.pdf))

4. **Thread-level vs. warp-level row assignment — pick by row length and level width.** Warp-per-row wastes lanes on short rows; thread-per-row serializes long rows. CapelliniSpTRSV's thread-level assignment gives 5.26× over prior warp-level sync-free on matrices with many components per level; DaCP identifies the regimes precisely: thread-level when rows are short and levels are wide (high components-per-level), warp-level (with warp-shuffle reduction of partial products) when rows are long. Performance between the two variants varies by up to 845× across matrix structures, so the analysis phase should record row-length statistics on the descriptor and select the variant per matrix — the AG-SpTRSV result (2.12–3.99× over cuSPARSE on A100/3080 Ti) is essentially this selection done well. ([CapelliniSpTRSV](https://dl.acm.org/doi/10.1145/3404397.3404400), [DaCP](https://link.springer.com/chapter/10.1007/978-981-96-2830-8_3), [AG-SpTRSV](https://dl.acm.org/doi/10.1145/3674911))

5. **Keep synchronization and partial sums in shared memory via subdomain/tile decomposition.** Partition the matrix so each subdomain's solution slice fits in one block's shared memory (~99 KB usable per block on sm_86 (opt-in); ~24k fp32 or ~12k fp64 values), turning inter-block synchronization into intra-block producer-consumer signaling and eliminating irregular global traffic for the vector — reported 10.7× triangular-solve speedup on structured problems, with flag traffic reduced 10–30× vs. global-memory synchronization. TileSpTRSV's variant (2D tiles, tile metadata stored as 8-bit rather than 32-bit ints) reports 5.29× over cuSPARSE with 8×8–32×32 tiles the empirical sweet spot on small-L2 consumer parts. Best for matrices with block/spatial structure (PDE grids); for fully irregular patterns fall back to practices 3–4. ([fine-grained domain decomposition](https://arxiv.org/html/2508.04917v1), [TileSpTRSV](https://link.springer.com/article/10.1007/s42514-023-00151-1))

6. **Shorten the critical path in the analysis phase with graph coloring.** When the level schedule is deep, apply approximate greedy/Luby coloring (on-GPU, O(rows) color map) during analysis to reorder rows into a few large independent sets — e.g. collapsing thousands of levels of ≤1000 rows into ~9 colors of hundreds of thousands of rows (6× on G3_circuit). The reorder is a one-time cost that lives on the descriptor, is deterministic once computed, and composes with either execution scheme. Only applicable where a reordered solve is acceptable to the caller (e.g. our own factorizations), since it changes the effective system ordering. ([NVIDIA graph-coloring blog](https://developer.nvidia.com/blog/graph-coloring-more-parallelism-for-incomplete-lu-factorization/))

7. **Offer an approximate iterative escape hatch for preconditioner duty.** Where SpSM is used as a preconditioner apply (not an exact solve), a truncated Jacobi series / two-stage Gauss–Seidel with K = 1–3 inner Jacobi iterations replaces the dependency-ordered solve with fully parallel SpMVs — no scheduling, deterministic by construction, and often faster on GPUs despite extra FLOPs because it decouples cost from DAG depth. Trade-off: outer Krylov iteration counts can rise, so gate it behind an explicit opt-in flag rather than a silent fallback. ([Jacobi-based approximate inverse](https://arxiv.org/pdf/2106.12836), [AMGCL two-stage smoothing](https://arxiv.org/pdf/1811.05704))

8. **Tensor cores only for wide-RHS fp32 dense blocks.** With k ≥ 8 RHS columns, the per-level update becomes a small dense GEMM that sm_86 tensor cores can accelerate at fp32/TF32; below that, launch/setup overhead dominates, and by our f32/f64 policy the fp64 path stays on CUDA cores regardless (fp64 runs at 1/64 rate on this part — do not build an fp64 TC path for sm_86). Treat this as a late-stage optimization after 1–5 are in place. ([Ampere architecture in-depth](https://developer.nvidia.com/blog/nvidia-ampere-architecture-in-depth/))

**Deterministic path.** The barriered level-set schedule is deterministic by construction: level ordering is fixed by the cached analysis, and within a level each row is written by exactly one thread group with no atomics — so the deterministic variant is simply the level-set kernel with warp-shuffle (not atomic) reductions for long rows. If the fast path uses sync-free atomics, the deterministic variant replaces atomic accumulation with producer-consumer flag signaling (one producer per row segment, completion signaled via volatile flag/fence), at a reported ~15–30% throughput cost but bitwise-reproducible results. Graph-coloring reorders and the iterative escape hatch (practice 7) are deterministic as-is. Both variants must run on the current stream with no host syncs; only the schedule differs, so both share the same cached descriptor.

## coo2csr (fused sort + compress)

**What dominates performance.** This op is pure data movement: a key–value sort of `nnz` packed index keys plus a compress pass, with essentially zero arithmetic. On the RTX A1000 (~192 GB/s, small L2) the cost model is simply *(number of full passes over nnz) x (bytes per element)* — every avoided radix pass and every eliminated intermediate round-trip to global memory is a directly measurable win. The small L2 means intermediate sorted-COO buffers will not stay cached between kernels, so fusion and pass-count reduction matter more here than on datacenter parts. The secondary regime is the common case where input is already (nearly) row-sorted — there the sort itself is wasted work and detection/early-exit dominates.

1. **Pack (row, col) into a single 64-bit key and sort values along with it.** Sort one `uint64` key (`row << 32 | col`) with the payload attached, rather than sorting an index permutation and gathering afterwards — the packed approach reorders data with coalesced scatters internally, while the permutation approach pays a non-coalesced gather pass that scales worse. This collapses the lexicographic (row, col) sort into a single stable radix sort and keeps workspace at O(nnz). Since we are int32-native, both indices always fit. ([GPU Multisplit, arXiv:1701.01189](https://arxiv.org/pdf/1701.01189))

2. **Narrow the radix bit range to the actual index widths.** With the packed key, only `ceil(log2(nrows)) + ceil(log2(ncols))` bits are meaningful; pass `begin_bit`/`end_bit` (CUB `DeviceRadixSort` semantics) so each skipped 8-bit digit removes a full pass over nnz. CUB documents ~10–20% overhead reduction from bit-subrange tuning, and for typical matrices (e.g. 20-bit row indices) this halves the pass count of a naive 64-bit sort. Compute the bit widths on host from `nrows`/`ncols` — no device sync needed. ([CUB DeviceRadixSort docs](https://nvidia.github.io/cccl/cub/api/structcub_1_1DeviceRadixSort.html))

3. **Detect already-sorted input cheaply and skip the sort.** Many callers hand us row-major COO already; run a warp-shuffle predicate pass (each thread compares its key against its neighbor's, ballot-reduce) — sorted input needs O(N) comparisons versus O(N log N) for a blind sort, and the check costs one read pass. Decision rule: if the scan finds zero inversions, jump straight to the compress phase; if nearly sorted, a merge-path style sort beats radix; only fully unsorted input takes the radix path. ([Modern GPU mergesort](https://moderngpu.github.io/mergesort.html), [Kepler shuffle pro-tip](https://developer.nvidia.com/blog/cuda-pro-tip-kepler-shuffle/))

4. **Fuse the compress into the sort's final scatter pass.** Emitting sorted COO to global memory and then launching a separate row-compress kernel adds a full read+write round-trip over nnz; instead, detect row boundaries in the last radix pass while the sorted tile is still in shared memory/registers and write `crow_indices` directly. NVIDIA's fusion case study shows the shape of the win (memory traffic 3 GB → 1 GB, 3x) — on a 192 GB/s part this is our headline edge over `thrust::sort` + `cusparseXcoo2csr`, which necessarily round-trips. ([Kernel fusion in CUDA](https://developer.nvidia.com/blog/kernel-fusion-in-nvidia-cuda-optimizing-memory-traffic-and-launch-overhead))

5. **For the compress phase, choose histogram+scan vs run-length-encode by input path.** On the sorted/early-exit path, row compression is exactly a run-length encode of the row indices — CUB `DeviceRunLengthEncode` is linear-work and fast on sorted data. On the full-sort path, prefer the histogram route: count rows into privatized shared-memory histograms during an early radix pass (shared atomics are hardware-native on Ampere and ~2x faster than global atomics), then one exclusive scan over `nrows` yields `crow_indices` with O(rows) workspace — the row counts fall out of the sort's own digit histograms when row bits occupy the high digits. ([CUB DeviceRunLengthEncode](https://nvidia.github.io/cccl/cub/api/structcub_1_1DeviceRunLengthEncode.html), [Shared-atomics histograms](https://developer.nvidia.com/blog/gpu-pro-tip-fast-histograms-using-shared-atomics-maxwell/))

6. **Custom Onesweep radix only above a size threshold; CUB below it.** Onesweep (single-pass per digit with decoupled look-back) cuts global traffic from 3n to 2n per digit and measured ~1.5x over CUB `DeviceRadixSort` on large inputs (29.4 GKey/s vs ~20 GKey/s at 256M keys on A100); the research pegs the payoff regime at nnz > ~10M. Decision rule: below that, CUB with bit-subrange tuning is the baseline and the engineering time is better spent on items 2–5; above it, Onesweep with decoupled look-back is the fast path. Look-back also suits sm_86's 16-blocks/SM limit since blocks proceed without chaining on their immediate predecessor. ([Onesweep, arXiv:2206.01784](https://arxiv.org/pdf/2206.01784), [AMD GPUOpen radix sort](https://gpuopen.com/learn/boosting_gpu_radix_sort/))

7. **Tune keys-per-thread (VT) to odd values; expect lower VT for 64-bit keys.** Odd VT (5, 7, 11…) sidesteps 32-way shared-memory bank conflicts; higher VT amortizes sync overhead but costs registers/shared memory and occupancy. Guidance from Modern GPU: 256-thread blocks with VT=11 for 32-bit keys, dropping toward VT=5 for 64-bit keys — benchmark VT ∈ {5, 7, 9, 11, 13} on the A1000 and prioritize occupancy (256–512 threads/block) for latency hiding on this bandwidth-starved part. ([Modern GPU performance](https://moderngpu.github.io/performance.html))

8. **Stage tiles with `cp.async` double-buffering.** Ampere's async global→shared copy lets the next tile of keys stream in while the current tile is being ranked/scattered; sm_86's 100 KB shared memory per SM comfortably fits two tiles at the VT sizes above. This hides load latency without extra occupancy, which matters when 64-bit keys already pressure registers. ([Ampere tuning guide](https://docs.nvidia.com/cuda/ampere-tuning-guide/index.html))

9. **Skewed row distributions: consider hybrid radix-then-local-comparison.** For power-law matrices, radix-partition by row bits first, then comparison-sort columns within each row bucket — the hybrid measured 2.32x on uniform and 1.66x on skewed data versus baseline. Decision rule: pure packed-key radix for uniform nnz-per-row; the hybrid only pays when a per-row histogram is being computed anyway (item 5) and rows are short enough to sort in shared memory. Treat as a v2 optimization, not the first implementation. ([Hybrid radix sort, arXiv:1611.01137](https://arxiv.org/pdf/1611.01137))

10. **Bound sort workspace with a circular buffer.** The look-back tail iterator guarantees data outside the active window is dead, allowing a fixed-size circular buffer (~2 MB in the reference implementation) instead of a second full nnz-sized ping-pong buffer. Useful on the A1000's limited VRAM and keeps us safely inside the O(nnz) workspace budget even with the payload attached. ([AMD GPUOpen radix sort](https://gpuopen.com/learn/boosting_gpu_radix_sort/))

**Deterministic path.** LSD radix sort is stable and bitwise-deterministic by construction, so the sort itself needs no variant — determinism concerns concentrate in the compress phase. Deterministic path: atomic-free per-block privatized histograms reduced across blocks, followed by an exclusive prefix scan (the CUB `agent_radix_sort_histogram` pattern); this also *wins* outright at small nnz where atomic contention isn't amortized. Fast path: shared-memory atomics within blocks, globally reduced — and where the input is clustered by row, aggregate runs in registers first (scan row indices within the VT-element tile, emit one atomic per run instead of one per boundary). Note that integer `atomicAdd` on counts yields deterministic *values* even when scheduling varies, so the fast/deterministic split here is cheap to maintain. ([CUB histogram agent](https://github.com/NVIDIA/cub/blob/main/cub/agent/agent_radix_sort_histogram.cuh), [Parallel histogram pattern](https://ajdillhoff.github.io/notes/gpu_pattern_parallel_histogram/))

## Grouped GEMM (segment_mm / gather_mm)

**What dominates performance.** On the RTX A1000 this op is almost never compute-bound: individual GEMMs are small, so the kernel lives or dies on (1) launch/scheduling overhead amortization across many problems, (2) load imbalance — wave-quantization tails where a few CTAs grind on large-K segments while the rest of the GPU idles, and (3) memory traffic against ~192 GB/s of bandwidth and a small L2, which makes every avoided global-memory round-trip (materialized gather operands, separate epilogue kernels, re-fetched B matrices) directly visible in the runtime. The acceptance bar (cublasGemmGroupedBatched) itself wins by ~1.2x over naive batched looping purely from dispatch amortization, so the scheduler design matters as much as the inner tile.

1. **Persistent kernel with a work-stealing tile loop.** Launch one grid sized to the GPU; each CTA loops, pulling (problem_idx, tile_idx) work items from a shared counter until the tile list is exhausted, instead of one launch (or one CTA wave) per problem. This eliminates both per-GEMM launch overhead and the wave-quantization tail from variable segment sizes — PyTorch's Triton persistent grouped GEMM measured 2.62x over a manual loop, and cuBLAS's grouped API gets 1.2x over batched looping from dispatch amortization alone. This is the standard design for many-small-GEMM workloads; treat multi-launch as obsolete for this op. ([PyTorch blog](https://pytorch.org/blog/accelerating-moes-with-a-triton-persistent-cache-aware-grouped-gemm-kernel/), [cuBLAS grouped GEMM](https://developer.nvidia.com/blog/introducing-grouped-gemm-apis-in-cublas-and-more-performance-updates/))

2. **Fuse the gather into the GEMM prologue; never materialize the gathered operand.** For gather_mm, read row indices in the prologue and issue the global→shared loads through them directly — CUTLASS Example 59 shows a gather-fused mainloop on Ampere (RTX 3080Ti) running at 32.0 TFLOP/s vs 29.9 for the dense kernel, i.e. fusion is essentially free when warp-level index access is coherent. Materializing costs an extra O(gathered_rows·K) write+read round-trip through GMEM (and would blow our O(nnz)/O(rows) workspace bound). Decision rule from the cost model: materialize only when the same index set is reused across multiple GEMMs (amortized gather) or operands are tiny (<8 KB); for segment_mm's contiguous segments and for one-shot gathers, fuse always. ([CUTLASS Ex. 59](https://github.com/NVIDIA/cutlass/blob/main/examples/59_ampere_gather_scatter_conv/README.md), [fused gather–GEMM–scatter cost model](https://arxiv.org/pdf/2604.18020v1))

3. **cp.async multi-stage pipelining to hide gather and load latency.** Use Ampere's `cp.async` (16-byte async global→shared copies) with double/multi-buffering: issue the next tile's (gathered) loads while the current tile computes. SonicMoE fuses token gather into the expert-GEMM mainloop exactly this way; well-structured cp.async pipelines sustain ~80–90% of peak bandwidth, which is the whole game at 192 GB/s. Practical depth on Ampere is 2–3 stages for our small tiles (SMEM cost scales linearly with stages; CUTLASS goes up to 5 only with large-tile configs). ([SonicMoE](https://arxiv.org/pdf/2512.14080), [HGEMM optimization notes](https://bruce-lee-ly.medium.com/nvidia-tensor-core-cuda-hgemm-advanced-optimization-5a17eb77dd85))

4. **Cache-aware tile launch ordering: schedule tiles sharing a B matrix consecutively.** In the persistent loop, order work items so all tiles of one segment (which reuse the same per-segment B) run back-to-back across CTAs before moving on, rather than row-major across problems. The Triton persistent kernel measured 1.33x latency improvement and +60% L2 hit rate from this grouped ordering — and the A1000's L2 is far smaller than the H100's, so the reuse window is shorter and the ordering matters more, not less. ([PyTorch blog](https://pytorch.org/blog/accelerating-moes-with-a-triton-persistent-cache-aware-grouped-gemm-kernel/))

5. **Problem scheduling: K-descending sort, uniform M/N tiles, and host-precompute vs device-only.** Sort problems by K descending before building the tile list — CUTLASS reports ~30% improvement on imbalanced grouped workloads because expensive large-K tiles get spread across all SMs instead of clustering on a few CTAs. Keep M/N tile shapes uniform across all problems (pad segments up to the tile M; only K varies), which is what CUTLASS's grouped scheduler requires and what keeps the work-item math to a prefix sum. For the schedule itself: `kHostPrecompute` (host builds the tile→problem map) wins when tiles are small/low-arithmetic-intensity; `kDeviceOnly` (warp-parallel prefix sums on device) wins when compute dominates — profile both. Project constraint: if segment offsets live in a device tensor, host precompute would force a D2H sync in the hot path — in that case use the device-only scheduler unconditionally. ([CUTLASS grouped scheduler docs](https://docs.nvidia.com/cutlass/latest/media/docs/cpp/grouped_scheduler.html))

6. **Tune the register-blocked tile with the small-GEMM design rule: minimize SMEM, maximize registers.** For small/batched GEMMs the classic large-GEMM recipe inverts: keep SMEM footprint small so multiple CTAs fit per SM (occupancy hides latency), and spend registers on independent accumulators instead. Keep the current 2D register blocking (e.g. 8×8 thread tile = 64 independent accumulators, ~4 FMAs per SMEM load, enough ILP to cover Ampere's 4-cycle FMA latency) inside a block→warp→thread hierarchy (reference Ampere config: 64×128 block / 32×64 warp / 8×8 thread, 128 threads). Decision rule: if a segment's tile working set is large (K in the hundreds+), shift toward the large-GEMM regime — bigger SMEM staging, deeper cp.async pipeline, 1–2 CTAs/SM; for many tiny segments, stay register-heavy and high-occupancy. ([Dongarra et al., batched GEMM autotuning](https://www.netlib.org/utk/people/JackDongarra/PAPERS/performance-design-and-autotuning.pdf), [GEMM anatomy walkthrough](https://medium.com/@emmanuelalo52/anatomy-of-a-cuda-gemm-from-naive-kernels-to-outperforming-cublas-on-blackwell-c394b04b5995))

7. **128-bit vectorized access and SMEM bank-conflict elimination.** Load and store in float4/int4 (16-byte) transactions — the walkthrough measured a 3.7x reduction in C-writeback instruction count and ~7x effective-bandwidth improvement from coalescing, and on a 192 GB/s part every point of bandwidth efficiency is throughput. In SMEM, break conflicts with either row padding (+4 bytes/row on 32-wide tiles, costs a little SMEM) or index permutation/swizzle (free SMEM, more index math); the gather path's irregular loads make it worth checking conflicts in Nsight rather than assuming. ([GEMM anatomy walkthrough](https://medium.com/@emmanuelalo52/anatomy-of-a-cuda-gemm-from-naive-kernels-to-outperforming-cublas-on-blackwell-c394b04b5995), [HGEMM optimization notes](https://bruce-lee-ly.medium.com/nvidia-tensor-core-cuda-hgemm-advanced-optimization-5a17eb77dd85))

8. **Epilogue fusion instead of follow-on kernels.** Keep the accumulator in registers/SMEM through any post-op — scaling, bias, or the scatter write for the output — rather than launching a separate elementwise kernel; each fused post-op saves a full output round-trip through GMEM (CUTLASS's Epilogue Visitor Tree is the reference pattern, and PyTorch's grouped-GEMM template got a further ~5% just from epilogue fusion). On a bandwidth-starved part this is cheap, low-risk win. ([Colfax EVT writeup](https://research.colfax-intl.com/epilogue_visitor_tree/), [cuBLAS grouped GEMM](https://developer.nvidia.com/blog/introducing-grouped-gemm-apis-in-cublas-and-more-performance-updates/))

9. **Adaptive grid/block sizing for tiny workloads.** When total tile count is small (few segments, small M), a fixed large CTA shape leaves SMs idle: halve the block size and double the grid so every SM gets work — SGLang measured 1.78x at batch size 1 from exactly this. Pick the launch shape at op dispatch time from the tile count; it's a host-side branch, no sync needed. ([SGLang/NVFP4 analysis](https://huggingface.co/blog/apsys/blackwell-nvfp4-comparison))

10. **v2 — fp16/bf16 inputs with fp32 accumulation on tensor cores.** Halving input width halves mainloop traffic, which on this GPU is worth more than the extra FLOPs; use `mma.sync` with fp32 accumulators (register-based — Ampere has no Hopper-style TMA or tensor memory, so the sm_86 path is HMMA + cp.async staging). If quantized paths ever land: on sm_86 prefer native INT8 IMMA over FP8-as-storage-with-decode — measured 118.06 TOPS vs 50.63 TOPS on RTX 3090 Ti (2.3x decode penalty); FP8 storage only pays when weight memory footprint is the binding constraint. ([CUTLASS Ex. 57 patterns](https://github.com/NVIDIA/cutlass/blob/main/examples/57_hopper_grouped_gemm/57_hopper_grouped_gemm.cu), [FP8-as-storage on Ampere](https://amohan.dev/blog/2026/fp8-as-storage-imma-ampere/))

**Deterministic path.** The research didn't cover determinism directly, but grouped GEMM is naturally deterministic as long as each output tile is owned by exactly one CTA and there is no atomic reduction: the work-stealing counter in the persistent kernel only changes *which* CTA computes a tile, not the fp accumulation order within it, so the fast persistent design and the deterministic path can be the same kernel. The one thing to keep off the deterministic path is split-K with atomic partial-sum reduction — if large-K segments ever need split-K for balance, the deterministic variant must reduce partials in a fixed order (separate O(rows) workspace + ordered reduction) instead of `atomicAdd`.

## Cross-cutting: Ampere tuning, determinism, mixed precision

**What dominates performance.** On the RTX A1000 (sm_86, ~176 GB/s DRAM, 1.5 MB L2, 16 SMs), nearly every op in this library is memory-bandwidth-bound: sparse kernels sit well below the ~0.5 FLOP/byte roofline knee, so peak-achieving means saturating DRAM, not filling FMA pipes ([sparsity roofline](https://arxiv.org/pdf/2310.00496)). The second-order regime is atomic contention in scatter/segmented reductions — costly here because the small L2 and low bandwidth amplify serialized traffic. Occupancy and pipelining tricks only pay off in the minority of latency- or compute-limited kernels (e.g. grouped_gemm inner tiles); for the streaming majority, once memory throughput saturates, extra occupancy buys nothing ([moderngpu](https://moderngpu.github.io/performance.html)). fp64 runs at 1/64 rate on this part — treat fp64 paths as correctness-only, never a tuning target.

### Prioritized practices

1. **Establish the roofline bound before micro-optimizing.** Compute arithmetic intensity (FLOPs / bytes moved) per kernel and compare achieved GB/s against the 192 GB/s peak; kernels under ~0.5 FLOP/byte are memory-bound and should be tuned for coalescing, data reuse, and fewer bytes — not occupancy or ILP. If measured bandwidth is well below the roofline, the culprit is access pattern or cache misses, not compute. A bandwidth-limited kernel at 95%+ memory utilization is the victory condition; stop there. ([sparsity roofline](https://arxiv.org/pdf/2310.00496), [moderngpu](https://moderngpu.github.io/performance.html))

2. **Never ship atomic-only reductions; pre-reduce in-warp, then one atomic per group.** Pure per-thread `atomicAdd` reductions run ~2 orders of magnitude slower than structured reductions, while modern deterministic alternatives measured <0.5% slower than non-deterministic fast paths on V100 (7.8% on GH200) — so the fast path itself should already do warp/block-level tree reduction and emit one atomic per segment. Where nnz distribution is irregular, sort-then-reduce groups writers to the same destination so contended atomics collapse to one per subgroup — this matters more at 192 GB/s than on datacenter parts. ([arXiv 2408.05148](https://arxiv.org/html/2408.05148v3), [NVIDIA CCCL determinism](https://developer.nvidia.com/blog/controlling-floating-point-determinism-in-nvidia-cccl/))

3. **Deterministic reductions via fixed-order segmented scan (ModernGPU three-phase), not slow serial fallbacks.** Upsweep (sequential per-thread reduction) → cooperative segmented scan over carry-outs → downsweep is deterministic by construction because the reduction tree order is fixed, and it load-balances irregular segments. Benchmarks: preprocessed reduce-by-key ~2x faster than vanilla, +10-20% from special-casing large segments in CSR frontends. Decision rule: this shines for segments ≳1000 elements; for many tiny segments the scan overhead dominates and a simpler fixed-tree block reduction (CCCL run-to-run mode, "slightly slower than non-guaranteed") is the better deterministic path. Full GPU-to-GPU bitwise reproducibility (CCCL RFA, exponent-binned accumulation) costs 20-30% — only reach for it if cross-hardware reproducibility becomes a requirement. ([moderngpu segreduce](https://moderngpu.github.io/segreduce.html), [NVIDIA CCCL determinism](https://developer.nvidia.com/blog/controlling-floating-point-determinism-in-nvidia-cccl/))

4. **Compile for sm_86 explicitly and police registers with `__launch_bounds__` + `-Xptxas=-v`.** sm_86 differs from A100: 48 max warps/SM (not 64), 100 KB shared memory per SM (99 KB max per block), 64K registers per SM, 16 concurrent blocks max, and 2x FP32 ops/cycle. Capping threads at 32 registers can double resident blocks per SM (4→8); gains from occupancy plateau past ~70%, and chasing 100% typically backfires via spills — stay under ~120 registers/thread and verify with `-Xptxas=-v`. Use `__launch_bounds__(threads_per_block, min_blocks_per_sm)` on every production kernel so the compiler optimizes toward the intended occupancy. ([Ampere tuning guide](https://docs.nvidia.com/cuda/ampere-tuning-guide/index.html))

5. **Grid-stride loops as the default kernel skeleton.** Each thread processes multiple elements spaced by grid stride: scales to any nnz/rows without host-side grid math, keeps launches `torch.compile`-friendly, and lets you shrink thread count to relieve register/SMEM pressure while preserving coalescing (consecutive threads touch consecutive elements). Most valuable when the dataset far exceeds one full-occupancy grid or the kernel is register-limited — both common here. ([grid-stride loops](https://developer.nvidia.com/blog/cuda-pro-tip-write-flexible-kernels-grid-stride-loops/))

6. **`cp.async` pipelining — only for compute-tiled kernels, and only after measuring stalls.** Ampere's async GMEM→SMEM copies bypass registers (freeing them for occupancy) and can cut long-scoreboard stalls from 40-70% of cycles to <5% with 2-7 pipeline stages; but SMEM footprint scales linearly with stages and past a crossover the occupancy loss exceeds the pipelining gain — acute with only 100 KB SMEM/SM. Decision rule: applies to grouped_gemm-style tiled compute with a "load N+1 while compute N" structure; skip it for pure gather/scatter/data-rearrangement kernels with no compute to overlap. Profile scoreboard stalls before and after; keep it only if stalls drop and occupancy holds. ([cp.async on Ampere](https://oracore.dev/news/cuda-cp-async-ampere-hbm-latency-en), [Ampere data movement](https://developer.nvidia.com/blog/controlling-data-movement-to-boost-performance-on-ampere-architecture/))

7. **Use sm_86 native warp reductions for integer work.** `__reduce_add_sync` / `__reduce_min_sync` / `__reduce_max_sync` (and bitwise variants) handle 32-bit integer reductions — segment IDs, index bounds, flag voting — in one instruction; keep floating-point reductions on shuffle/shared-memory trees since the native instructions are integer-only. ([Ampere tuning guide](https://docs.nvidia.com/cuda/ampere-tuning-guide/index.html))

8. **L2 persistence for hot metadata, sized to the tiny L2.** Ampere access-policy windows can pin repeatedly-touched structures (CSR row pointers, segment offsets) in L2 while values stream through, with a measured ~20% latency win when the pinned set fits the reservation; the constraint is `hitRatio × num_bytes ≤ persistingL2CacheSize` to avoid thrashing. On the A1000's 1.5 MB L2 this only helps when metadata is a small fraction of that — measure, and skip for uniform-access kernels. ([L2 persistent cache](https://leimao.github.io/blog/CUDA-L2-Persistent-Cache/))

9. **Mixed precision: accumulate in fp32, avoid fp64 hot paths.** seglse and any future fp16/bf16 input paths must carry fp32 accumulators through the reduction tree (and through atomics on the fast path) — the determinism findings above apply to the fp32 accumulator ordering, not the storage dtype. With fp64 at 1/64 rate on this part, fp64 support stays correctness-only: same code paths, no fp64-specific tuning.

10. **Benchmark like the laptop GPU it is.** Lock clocks (`nvidia-smi -lgc`), idle the device 10+ s, run ≥10 warmup iterations, and report run-to-run variance — mobile Ampere throttles aggressively and the 1.5 MB L2 makes cache state visible between runs. Use `torch.utils.benchmark` (CUDA timing + warmup) for fast-vs-deterministic relative comparisons, and stage profiling as Nsight Systems for the timeline → Nsight Compute basic set (SpeedOfLight, Occupancy, LaunchStats) → detailed metrics, keeping the profiling configuration identical across compared runs. Interpretation pitfalls: a high memory-throughput % is not proof of efficiency (check achieved GB/s against the roofline and per-level L1/L2 hit rates — one case showed 87% L1 but <50% L2); warp-stall reasons are only meaningful once occupancy is healthy; a large achieved-vs-theoretical occupancy gap points to workload imbalance or a binding resource, diagnose that first. ([Nsight Compute guide](https://docs.nvidia.com/nsight-compute/ProfilingGuide/index.html), [kernel surgery](https://themlsurgeon.substack.com/p/performing-kernel-surgery-profiling), [NASA HECC ncu workflow](https://www.nas.nasa.gov/hecc/support/kb/performance-analysis-of-your-gpu-cuda-kernels-with-nsight-compute-cli_706.html))

### Deterministic path

`torch.use_deterministic_algorithms(True)` never silently falls back — ops either dispatch a deterministic implementation or raise `RuntimeError` ([PyTorch reproducibility](https://docs.pytorch.org/docs/stable/notes/randomness)). So every `torch.library` op here registers two explicit paths: the atomic fast path (with warp pre-reduction per practice 2) and a fixed-order path (segmented scan or fixed reduction tree per practice 3) selected when the deterministic flag is set; if a kernel's deterministic path is ever unimplemented, raise rather than degrade. The measured cost of run-to-run determinism on modern hardware is small (<0.5% on V100 for well-built variants; CCCL's fixed-tree mode "slightly slower") — the deterministic path should be a real implementation, not a penalty box. Determinism target is run-to-run on the same GPU; cross-GPU bitwise reproducibility (RFA, 20-30% overhead) is out of scope unless a requirement appears.

## PTX & microarchitecture layer (sm_86)

Honest framing first: for the memory-bandwidth-bound majority of this kernel set (SDDMM, seglse, SpMM/SpMV, coo2csr) this layer buys roughly a 5–20% margin on top of the algorithm-level work already documented above — worth taking, not worth redesigning around. There are two exceptions where it is decisive: **grouped GEMM**, where the cuBLAS-parity bar is still open and tile staging (`cp.async`), register discipline, and the tensor-core v2 path *are* the roadmap; and the **SpSM sync-free spin-flag path**, where PTX-level memory ordering is a correctness requirement, not a tuning knob — get the acquire/release scopes wrong and the kernel is racy, get the polling cost wrong and the sync-free path loses to level scheduling. Several sub-topics below came back thin from the research pass (fetch failures on NVIDIA doc sections); those are flagged rather than filled in with guesses.

### Cache control & prefetch

- **`ld.global` cache operators for selective caching** — `[SpMM/SpMV, SDDMM, seglse]`. PTX gives per-load cache control: `.nc` (non-coherent), `.cg` (cache at L2 only), `.ca` (cache at all levels, default), `.cv` (volatile). Syntax: `ld.global.cg.f32 r0, [addr];`, `ld.global.nc.u32 r1, [addr];`. Pattern for sparse kernels: stream once-touched values/column-indices with L1-bypassing operators so the reused structures (row pointers, segment offsets, dense-matrix panels) keep their L1/L2 residency. All operators supported on sm_86. Source: [PTX ISA 9.3, cache operators](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html). **Intrinsic mapping verified by local SASS probe (nvcc 13.0, sm_86, 2026-07-17):** `__ldcs` → `LDG.E.EF` (evict-first streaming), `__ldg`/`const __restrict__` → `LDG.E.CONSTANT` (read-only cache), `__stcs` → `STG.E.EF` — the intrinsics lower as advertised, no raw PTX needed for streaming loads/stores.
- **Rolling prefetch with `__pipeline_memcpy_async`** — `[SDDMM, SpMM]`. NVIDIA's recommended Ampere pattern: `__pipeline_memcpy_async()` + `__pipeline_commit()` + `__pipeline_wait_prior(N)` in a rolling loop; a prefetch distance of ~6 iterations gave ~60% speedup in NVIDIA's Ampere (A100) measurements. Avoids the barrier-forced early completion of naive prefetch. Same compute capability family as GA107, but re-tune the distance on the A1000 — 16 SMs and 176 GB/s is a very different latency/bandwidth balance than A100. Source: [NVIDIA blog: GPU memory prefetching](https://developer.nvidia.com/blog/boosting-application-performance-with-gpu-memory-prefetching/).
- **L2 persistent window for row pointers** — `[SpMM/SpMV, SpSM, seglse]`. cc 8.0+ `cudaStreamAttrValue::accessPolicyWindow`: `base_ptr` = row-pointer buffer, `hitProp = cudaAccessPropertyPersisting`, `missProp = cudaAccessPropertyStreaming`, `hitRatio` tuned so `hitRatio × num_bytes` stays under `cudaLimitPersistingL2CacheSize`. On the A1000's 1.5 MB L2 a ~256 KB reservation holds ~64K 32-bit row pointers while values stream through the remaining L2. Directly complements the SpSM level-schedule descriptor cache. Source: [Lei Mao: CUDA L2 persistent cache](https://leimao.github.io/blog/CUDA-L2-Persistent-Cache/). *Thin spot:* PTX-level `createpolicy`/`applypriority`/`evict_last` hints weren't covered by any successful fetch — the stream-attribute API is the verified route.

### cp.async details

- **Cache modifiers: `.ca` vs `.cg`** — `[grouped GEMM, SpMM]`. `cp.async.ca` stages through L1; `cp.async.cg` bypasses L1, going L2→shared directly — 16-byte transfers only for `.cg`. Use `.cg` for one-way tile traffic (global→shared, never re-read), which is exactly the grouped-GEMM tile pattern; the HGEMM reference implementation uses `.cg` on 16-byte transfers with L2 prefetch hints (`.L2::128B` etc.). Use `.ca` only when the same lines are re-read through L1. Source: [Bruce Lee HGEMM optimization](https://bruce-lee-ly.medium.com/nvidia-tensor-core-cuda-hgemm-advanced-optimization-5a17eb77dd85).
- **Transfer sizes 4/8/16 bytes** — `[grouped GEMM]`. `cp.async` moves 4, 8, or 16 bytes per instruction; prefer 16-byte where alignment permits (fewer instructions, full bandwidth on a 176 GB/s part), fall back to smaller sizes at unaligned group boundaries. Source: HGEMM article above; PTX ISA 9.3.
- **`cp.async.commit_group` / `cp.async.wait_group N`** — `[grouped GEMM]`. Batch pending copies into a group, compute, then `wait_group` to bound in-flight groups — the primitive under 2–3-stage pipelines. The fetch confirmed these instructions exist (PTX ISA §9.7.9.26.3.x) but did not retrieve full semantics; the `cuda::pipeline` / `cooperative_groups::memcpy_async` C++ wrappers were likewise TOC-only. Verify exact wait semantics against the PTX ISA before writing raw PTX; the C++ wrapper is the safer first implementation.
- **Zero-fill for edge tiles** — `[grouped GEMM, SpMM]`. `cp.async` supports predication with zero-fill for partial tiles (out-of-bounds lanes write zeros to shared), eliminating separate boundary kernels — attractive for grouped GEMM's ragged per-group dimensions. *Thin:* the exact `-zfill`/`ignore-src` syntax was not in any fetched excerpt; confirm in PTX ISA §9.7.9.26 before use.
- **LDGSTS: zero-register tile staging** — `[grouped GEMM]`. `cp.async` compiles to the `LDGSTS` SASS instruction: global→shared without intermediate register staging ("does not use any registers" per the HGEMM article). This is the lever for the register-blocked grouped-GEMM tiles: bigger shared-memory staging (up to the 99 KB opt-in) without adding register pressure. Evidence is indirect (blog + inference, not a fetched SASS spec) — verify by inspecting the disassembly per the SASS checklist below.
- **Multi-stage staging for grouped GEMM** — `[grouped GEMM]`. While one stage computes, the next tile loads: 2–3 stages hide tile-load latency across the A1000's 48 warps/SM without explicit barriers between stages (commit/wait handles it). This is the direct successor to the current register-blocked tiles and the substrate the tensor-core v2 path will reuse. Source: HGEMM article (multi-stage buffering); [NVIDIA Ampere data-movement blog](https://developer.nvidia.com/blog/controlling-data-movement-to-boost-performance-on-ampere-architecture/).

### Memory ordering & spin-flags (SpSM)

The sync-free candidate path lives or dies here.

- **`ld.acquire` / `st.release` with explicit scope** — `[SpSM]`. Scopes: `.cta` (block, cheaper), `.gpu` (grid-wide), `.sys`. Row-completion signaling between blocks needs `.gpu`; block-internal elimination can use `.cta`. Release orders all prior writes (the solved row values) before the flag store; acquire orders the flag load before subsequent reads. `.cluster` scope is **sm_90+ only — N/A on sm_86**; the choice here is binary `.cta` vs `.gpu`. Source: [PTX ISA 9.3 §8 memory consistency model](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html).
- **`red.release` for completion counters** — `[SpSM]`. `red.add.release.gpu` (also `min`/`max`/`and`/`or`/`xor`) fuses the atomic counter update with release ordering in one instruction — preferred over a separate `st.release` + `atom` pair for the per-row dependency counters. Source: PTX ISA 9.3 §9.7.14.6.
- **`membar` vs per-instruction ordering** — `[SpSM]`. `membar.cta`/`membar.gl`/`membar.sys` order *all* prior memory ops — heavier than needed for point-to-point row signaling. Prefer scoped acquire/release on the specific flag; reserve `membar.gl` for cases where bulk visibility is genuinely required. Do not write new `volatile` + `membar` code — it's the legacy idiom the formal model replaced. Source: PTX ISA 9.3 §8, §9.7.14.4.
- **`__nanosleep()` backoff in the spin loop** — `[SpSM]`. Pattern: `while (!load_acquire(&flag)) __nanosleep(200);` with 100–1000 ns backoff. Cuts polling pressure on L2 and power draw — material when up to 768 warps (16 SMs × 48) could be polling. Available as a device intrinsic on sm_70+. Source: [CUDA C++ Programming Guide, device intrinsics](https://docs.nvidia.com/cuda/cuda-c-programming-guide/index.html).
- **The composed pattern** — `[SpSM]`. Producer: finish row solve → `red.add.release.gpu` on the row-done counter. Consumers: acquire-load poll with nanosleep backoff. The PTX memory-model axioms (coherence, atomicity, causality) are what make this race-free: release guarantees the row data is visible before the counter increment is. If the counter itself becomes the contention point, split to per-warp counters. Source: PTX ISA 9.3 §8 + Programming Guide.
- *Thin spot:* the portable C++ spelling — `cuda::atomic<T, cuda::thread_scope_device>` with `cuda::memory_order_acquire/release` (libcudacxx, CUDA 11.2+) — is the recommended way to write all of the above without raw PTX, but the libcudacxx docs page 404'd during research. Verify the current API path in CCCL docs before coding against it.

### Warp & atomic instructions

- **`redux.sync` — CONFIRMED available on sm_86 (local SASS probe, nvcc 13.0, 2026-07-17).** A compiled probe of `__reduce_add_sync`/`__reduce_max_sync` for sm_86 emits single `REDUX.SUM.S32` / `REDUX.MAX.S32` instructions (one research agent had claimed Hopper-only from a truncated fetch — wrong for the integer variants; float variants remain newer-arch-only). Use for 32-bit integer reductions: segment IDs, index bounds, flag voting. Float reductions stay on `__shfl_down_sync` trees — `[seglse, SpMV, SDDMM]`.
- **`match.sync` (`match.any`/`match.all`) for segment grouping** — `[seglse, coo2csr]`. Would let a warp discover which lanes share a segment key without a shuffle loop. *Thin:* the fetched PTX excerpt confirmed the instruction exists (§9.7.14.11) but not its sm_86 behavior; `__ballot_sync` + `__shfl_sync` are the fetch-verified equivalents. Source: [NVIDIA blog: voting and shuffling for atomics](https://developer.nvidia.com/blog/voting-and-shuffling-optimize-atomic-operations/).
- **Warp-aggregated atomics before global writes** — `[seglse, coo2csr, SpMM COO path]`. Shuffle-combine matching keys within the warp, then issue one atomic per unique key. Proven technique (Kepler/Maxwell-era blog, principles carry to Ampere); this is the PTX-level justification for the warp pre-reduction practice already mandated in the atomics sections above. Source: same blog.
- **Latency baselines for atomic cost models** — `[all]`. Measured on Ampere A100 (sm_80, closest published numbers): shared memory ~29 cycles, global memory ~466 cycles random-access. Use these as order-of-magnitude anchors for shared-vs-global atomic staging decisions on GA107, not as exact numbers — profile locally. Source: [arXiv:2402.13499](https://arxiv.org/abs/2402.13499) (Table IV).
- *Thin spots:* block-scoped atomics (`atomicAdd_block`) exist but no throughput data was retrievable; `atom.global.cas` retry-loop latency likewise unmeasured in any fetched source. Both need local microbenchmarks if the SpSM spin path ends up CAS-based.

### Compiler knobs

- **`-Xptxas -O{0..3}`** — `[all]`. ptxas optimization levels exist and are documented as flags, but the NVCC manual does not specify what each level changes (scheduling, unrolling, register allocation). Treat as an A/B knob: build the hot kernels at `-O3` (default) and `-O2` and diff the SASS + `-res-usage` numbers. Source: [NVCC compiler driver docs](https://docs.nvidia.com/cuda/cuda-compiler-driver-nvcc/index.html).
- **Whole-module cache-mode defaults: `-Xptxas -dlcm=<ca|cg|cs|cv>` / `-flcm`, `-dscm` / `-fscm`** — `[SpMM/SpMV, SDDMM]`. Set the default (or force) load/store cache operator module-wide — a zero-code-change way to test the L1-bypass hypothesis from the cache-control section before hand-annotating individual loads. Per-mode performance effects are not documented; measure. Source: NVCC docs.
- **`--use_fast_math` and the seglse exp question — open safety gap** — `[seglse]`. The flag exists and swaps `expf` for the fast intrinsic path, but the CUDA Math API ULP tables did not fetch, so **no source retrieved specifies the precision of fast-math `expf` — the safety of fast-math exp inside logsumexp is an open question, not a settled yes**. What we know structurally: after the max-shift, exp arguments lie in (-inf, 0] and outputs in (0, 1], so the question reduces to fast-exp's ULP error on that range plus its subnormal behavior near the underflow edge. Required before enabling: pull the ULP tables from the [CUDA Math API docs](https://docs.nvidia.com/cuda/cuda-math-api/index.html) and run the seglse accuracy suite (including power-law and near-degenerate segments) with and without the flag. Until then, seglse builds without `--use_fast_math`.
- **`-ftz=true` denormal flushing** — `[seglse, SpMM]`. Flushes subnormals to zero; a Fermi-era NVIDIA case study showed ~20% speedup with no accuracy loss *when denormals don't naturally occur*. seglse's shifted exponentials deliberately produce tiny values near the underflow boundary, so this needs the same empirical accuracy gate as fast-math — plausibly benign (flushed terms would contribute ≈0 to the sum) but that reasoning is ours, not the source's. Source: [NVIDIA blog: flush denormals with confidence](https://developer.nvidia.com/blog/cuda-pro-tip-flush-denormals-confidence/).
- *Thin spots:* `-Ofc` (fast-compile levels max/mid/min/0) exists but level semantics are unspecified — irrelevant for release builds, possibly useful for CI compile time. Per-kernel scoping of fast-math/pragma behavior is not documented anywhere fetched; assume compilation-unit granularity, which argues for keeping seglse in its own .cu file so its flags stay independent.

### Tensor-core PTX (v2 grouped GEMM)

- **`mma.sync.aligned.m16n8k16.f32.f16` (and `.bf16`, and `m16n8k8`)** — `[grouped GEMM v2]`. The sm_86 tensor-core primitive: 16×8 output tile, k=16 or 8, fp16/bf16 inputs with **fp32 accumulators** — the accumulate-in-fp32 rule from the mixed-precision section is free here. Fully supported on sm_86. Source: [PTX ISA 9.3](https://docs.nvidia.com/cuda/parallel-thread-execution/index.html).
- **`ldmatrix.sync.aligned.m8n8.x{1,2,4}[.trans]`** — `[grouped GEMM v2]`. Loads 1/2/4 8×8 fragments from shared memory into the warp's registers in one instruction, with optional transpose-on-load — the bridge between cp.async-staged tiles and mma operands. Supported on sm_86. Source: PTX ISA 9.3.
- **Fragment register layouts: read the tables, don't trust summaries** — `[grouped GEMM v2]`. Operands are distributed across the warp's registers in fixed per-thread layouts specified in the PTX ISA mma section. The research pass returned per-fragment register counts that don't self-check (e.g. 8 registers for an m16n8k16 f16 A-fragment where the arithmetic gives 4×`.b32` per thread), so take layouts directly from the PTX ISA tables when writing the kernel.
- **Bank-conflict elimination: padding or permuted (XOR/swizzled) layouts** — `[grouped GEMM v2]`. 32 banks × 4 bytes on sm_86; naive power-of-2 tile strides serialize `ldmatrix` accesses. Either pad the shared-memory tile stride (+1 element past the natural width) or permute the store pattern; the HGEMM reference shows both work. Source: [HGEMM article](https://bruce-lee-ly.medium.com/nvidia-tensor-core-cuda-hgemm-advanced-optimization-5a17eb77dd85).
- **The full v2 recipe is proven on Ampere** — `[grouped GEMM v2]`. cp.async multi-stage prefetch + `ldmatrix` + `mma.sync` + wide (`float4`/`int4`) global loads reaches 95%+ of cuBLAS (sometimes exceeding it) on Ampere-class GPUs (RTX 3090/A6000) across sizes 256–16384 — evidence the cuBLAS-parity bar is reachable with this exact stack, modulo grouped/ragged bookkeeping. Source: HGEMM article.
- *Thin spot:* tensor-core latency/throughput microbenchmark data (arXiv:2206.02874), CUTLASS Ampere specifics, and the GA10x whitepaper all failed to fetch. Instruction latencies and CUTLASS's grouped-GEMM scheduling remain unresearched — revisit before v2 design lock, alongside the Stream-K caveat in the addendum above.

### SASS audit workflow (checklist)

Run this on every kernel before declaring it done; it is how several claims above (LDGSTS, vectorization, sync-free) get verified rather than assumed. Sources: [CUDA Binary Utilities](https://docs.nvidia.com/cuda/cuda-binary-utilities/index.html), [Nsight Compute Profiling Guide](https://docs.nvidia.com/nsight-compute/ProfilingGuide/index.html), [siboehm CUDA MMM](https://siboehm.com/articles/22/CUDA-MMM).

1. **Build with visibility:** add `-Xptxas -v` (register/smem/spill stats to stderr) and keep line info (`-lineinfo`) for source correlation.
2. **Check resources:** `cuobjdump -res-usage kernel.cubin` — per-function `REG`/`STACK`/`SHARED`. **`STACK > 0` means register spills**; on bandwidth-bound kernels every spilled access is wasted DRAM budget. Cross-check against the `lmem` line from `-Xptxas -v`.
3. **Disassemble:** `cuobjdump -sass`, then `nvdisasm -g` (source annotations), `-plr` (register liveness — find registers held live across long ranges for no reason), `-cfg` (control-flow graph as DOT).
4. **Verify vectorization:** grep the SASS for `LDG.E.128` / `STG.E.128` / `LDS.128`. Runs of scalar 32-bit `LDG.E` where a 128-bit load was expected mean a missing alignment guarantee or `float4` cast. Any `LDL`/`STL` is a spill — go back to step 2.
5. **Verify cp.async lowering** (grouped GEMM): confirm the tile loads became `LDGSTS`, not load-then-store through registers.
6. **Audit addressing overhead** (sparse indexing paths): look for long `IMAD.WIDE` dependency chains computing row/col offsets (IMAD ≈ 4 cycles on Ampere); chains of 10+ that gate the `LDG`s need hoisting or strength reduction.
7. **Verify the SpSM sync-free path structurally:** `nvdisasm -cfg`, confirm the spin loop contains **no `BAR` instructions** — barrier-free is the entire point; a stray `__syncthreads` lowered into the loop silently converts it back to a synchronized kernel.
8. **Correlate stalls to source:** Nsight Compute with the Source Counters section → Source page; sampled warp-stall reasons map to SASS lines. Watch `l1tex__*` (L1), `lts__*` (L2), `launch__registers_per_thread_allocated`, `launch__occupancy_per_register_count` (is register pressure the occupancy limiter?).
9. **Iterate the occupancy trade:** if register-limited, A/B `__launch_bounds__(threads, minBlocks)` and `-maxrregcount` against measured runtime, not occupancy percentage — the profiling section above already establishes that occupancy is a means, not the target.

## Gap-fill research addendum (2026-07-17)

- **[grouped_gemm]** *Stream-K work decomposition.* Instead of tiling only M×N, Stream-K splits the K (reduction) dimension across CTAs so trailing CTAs pick up K-segments of existing tiles rather than idling — eliminates wave-quantization tails when tile count isn't a multiple of SM count. Payoff is data-dependent on small consumer parts; highest when `(M×N tiles) % SM_count ≠ 0`. Source: Osama, Merrill, Garland et al., *Stream-K*, PPoPP 2023 — https://dl.acm.org/doi/10.1145/3572848.3577500

- **[grouped_gemm]** *Partial-tile fixup cost.* Multi-CTA K-segments on one output tile need a separate reduction pass; workspace is O(ctas_per_tile × tile_m × tile_n) — bounded on consumer GPUs. Deterministic fixup requires ordered accumulation (extra launch + sync, measurable on small problems); the atomic variant is faster but non-deterministic. Sources: Stream-K paper (above); CUTLASS — https://github.com/nvidia/cutlass

- **[grouped_gemm]** *Caveat:* search budget ran out before verifying current CUTLASS API status for Stream-K + grouped GEMM on sm_86 or getting consumer-GPU (RTX A1000-class) benchmark numbers. Treat the above as design direction, not measured wins; verify against the CUTLASS examples/profiler before committing.

- **[seglse_bidir]** *Cached-CSC-permutation alternative to atomics — design note (unsourced; web-search budget exhausted before this angle ran; settle by benchmark).* The spec already mandates a deterministic path under `torch.use_deterministic_algorithms(True)`, and the only known deterministic column-direction strategy is exactly this: cache a CSR-order → column-sorted permutation per sparsity pattern (same descriptor-cache pattern as the SpSM level-schedule) and run a second sorted segmented pass. So the permutation path must be built regardless — the open question is only whether the atomic fused single-pass earns its keep as the fast path. Framing for the benchmark: the fused pass saves one O(nnz) traversal of values/indices (~8–16 bytes/nnz of DRAM traffic) but pays float-atomic-max + CAS-rescale per entry on the column accumulators, which degrades badly on power-law column degrees (hot-column contention serializes); the two-pass permuted variant costs the extra traversal plus a gather through the permutation (extra 4–8 bytes/nnz) but is contention-free and bitwise-reproducible. On a ~176 GB/s part the extra pass costs roughly nnz×12 B / 176 GB/s — benchmark the crossover on skewed-column suites before locking the design, and record which path each benchmark row used.

- **[coo2csr]** *Deterministic dedup pipeline: stable radix sort → CUB ReduceByKey.* Pack (row‖col) into a 64-bit key, one DeviceRadixSort pass, then `cub::DeviceReduce::ReduceByKey` (BlockDiscontinuity segment heads + scan). Fixed sort order makes FP sums bitwise reproducible — no atomics. Rough RTX A1000 budget: sort + one reduce pass ≈ 1–2 ms per 1M nnz at ~192 GB/s peak. Sources: CUB — https://github.com/NVIDIA/cccl/tree/main/cub; CUDA C Programming Guide — https://docs.nvidia.com/cuda/cuda-c-programming-guide/

- **[coo2csr]** *Clean-data fast path.* A single warp-shuffle adjacent-key-equality pass (à la `DeviceRunLengthEncode`) detects whether any duplicates exist; skip the whole reduce-by-key pass when the COO input is already coalesced — the common case. Source: CUDA C Programming Guide (shuffle) + CUB DeviceRunLengthEncode docs — https://github.com/NVIDIA/cccl/tree/main/cub

- **[coo2csr]** *Non-deterministic alternative: warp-aggregated atomics.* One atomic per unique key per warp; ~3–5x faster when duplicates cluster, but accumulation order is timing-dependent even on sorted keys — violates the determinism bar, keep only as an opt-in path if at all. Source: NVIDIA Cooperative Groups blog — https://developer.nvidia.com/blog/cooperative-groups/

- **[coo2csr]** *Workspace shape.* CUB/Thrust reduce-by-key needs O(nnz) temp scratch (no library primitive is O(rows)-bounded); a rows-bounded budget requires a custom block-per-row kernel or multi-pass segmented reduce. PyTorch instead defers coalescing entirely (concat, coalesce only when nnz exceeds a threshold) — a good model for gradient accumulation, but inline reduce-by-key is right for the one-shot COO→CSR step. Sources: CUB temp-storage docs — https://github.com/NVIDIA/cccl/tree/main/cub; PyTorch `SparseCUDATensorMath.cu` — https://github.com/pytorch/pytorch/blob/main/aten/src/ATen/native/sparse/cuda/SparseCUDATensorMath.cu

- **[coo2csr]** *cuSPARSE does NOT dedup.* `cusparseXcoo2csr` is format conversion only; docs warn correctness is not ensured with duplicate column indices in a row. Deduplication must happen upstream — hard requirement, not an optimization. Source: cuSPARSE docs — https://docs.nvidia.com/cuda/cusparse/index.html
